# 多账号认证

## 1、需求场景
有时候，我们会在一个项目中设计两套账号体系，比如一个电商系统的 `user` 端和 `admin` 端，在这种场景下，如果两套账号我们都使用`StpUtil`的API进行登录认证，那么势必会发生登录冲突。

在Sa-Token中，这个问题的模型叫做：多账号体系认证。

要解决这个问题，我们必须有一个合理的机制将这两套账号的授权区分开，让它们互不干扰执行。

---

## 2、演进思路
假设我们的user表和admin表都有一个id=10001的账号，它们的登录代码：`StpUtil.login(10001)` 是一样的，那么问题来了：在`StpUtil.getLoginId()`获取到的账号id如何区分它是user用户，还是admin用户？

你可能想到的是给一个固定前缀，比如`StpUtil.login("user_" + 10001)`、`StpUtil.login("admin_" + 10001)`，这样确实是可以解决问题的，但是同样的：你需要在`StpUtil.getLoginId()`时再裁剪掉相应的前缀才能获取真正的账号id，这样一增一减就让我们的代码变得无比啰嗦。

那么，有没有从框架层面支持的，更优雅的解决方案呢？

---

## 3、解决方案
前面几章的api调用，都是经过`StpUtil`类的各种静态方法进行授权认证，而如果我们深入它的源码，点击溯源就会发现，此类其实没有任何代码逻辑，唯一做的事就是对成员变量 `stpLogic` 的各个API包装一下进行转发。

这样就有两个好处：
- `StpLogic`类的所有函数都可以被重写，按需扩展。
- 在构造方法时随意传入一个不同的 `loginType`，就可以再搞一套账号登录体系。

---

## 4、操作示例
比如说，对于原生`StpUtil`类，我们只做`admin`权限认证，而对于`user`用户，我们则：
1. 新建一个新的权限认证类，比如：`StpUserUtil.java`。
2. 将`StpUtil.java`类的全部代码复制粘贴到 `StpUserUtil.java` 里。
3. 更改一下其 `LoginType`，比如：

```java
public class StpUserUtil {

    // 账号体系标识
    public static final String TYPE = "user";    // 将 LoginType 从"login"改为"user"

    // 其它代码 ...

}
```

成品样例参考：码云：`StpUserUtil.java`

4. 接下来就可以像使用 `StpUtil` 一样的方式，调用 `StpUserUtil` 上的所有方法了，这两套账号认证的逻辑是完全隔离的。例如：

```java
// 可以在 任意地方 调用 StpUserUtil 上的方法
StpUserUtil.login(10001);        // 在当前会话 登录user账号 10001
StpUserUtil.getLoginId();        // 获取当前会话登录的user账号id
StpUserUtil.getSession();        // 获取当前user账号的 Session 对象
StpUserUtil.checkPermission("xx");    // 校验当前登录的 user 账号是否具有 xx 权限
```

## 5、Kit模式
如果你觉得“复制代码”的方式繁琐不够优雅，这里还有另一种方案：建立一个 `StpKit.java` 门面类，声明所有的 `StpLogic` 引用：

```java
/**
 * StpLogic 门面类，管理项目中所有的 StpLogic 账号体系
 */
public class StpKit {

    // 默认原生会话对象
    public static final StpLogic DEFAULT = StpUtil.stpLogic;

    // Admin 会话对象，管理 Admin 表所有账号的登录、权限认证
    public static final StpLogic ADMIN = new StpLogic("admin");

    // User 会话对象，管理 User 表所有账号的登录、权限认证
    public static final StpLogic USER = new StpLogic("user");

    // XX 会话对象，（项目中有多少套账号体系，就声明几个）
    public static final StpLogic XXX = new StpLogic("xx");

}
```

在需要登录、权限认证的地方：

```java
// 在当前会话登录 Admin 账号 10001
StpKit.ADMIN.login(10001);

// 在当前会话登录 User 账号 10001
StpKit.USER.login(10001);

// 校验当前会话是否以 Admin 账号登录，并具有 article.add 权限
StpKit.ADMIN.checkPermission("article.add");
// 校验当前会话是否以 User 账号登录，并通过了二级认证
StpKit.USER.checkSafe();

// 获取当前 User 会话的 Session 对象，并进行后续操作
StpKit.USER.getSession().set("name", "zhang");
```

## 6、在多账户模式下使用注解鉴权
框架默认的注解鉴权 如`@SaCheckLogin` 只针对`StpUtil`进行鉴权。

例如，我们在一个方法上加上`@SaCheckLogin`注解，这个注解只会放行通过`StpUtil.login(id)`进行登录的会话，而对于通过`StpUserUtil.login(id)`进行登录的会话，则始终不会通过校验。

那么如何在注解上，要鉴权的是你自定义的`StpUserUtil`，而不是原生`StpUtil`？很简单，你只需要指定一下注解的type属性即可：

```java
// 通过type属性指定此注解校验的是我们自定义的StpUserUtil，而不是原生StpUtil
@SaCheckLogin(type = "user")
@RequestMapping("info")
public String info() {
    return "查询用户信息";
}
```

注：`@SaCheckRole("xxx")`、`@SaCheckPermission("xxx")` 同理，亦可根据type属性指定其校验的账号体系，此属性默认为 `""`，代表使用原生`StpUtil`。

## 7、使用注解合并简化代码
交流群里有同学反应，虽然可以根据 `@SaCheckLogin(type = "user")` 指定账号类型，但几十上百个注解都加上这个的话，还是有监察局，代码也不够优雅，有没有更简单的方案？

我们期待一种：`@SaUserCheckLogin` 的能力，自定义一个注解，标注上 `@SaCheckLogin(type = "user")`，然后在方法上标注这个自定义注解，效果等同于标注上了原注解。

很遗憾，JDK默认的注解处理并没有提供这种【注解合并】的能力。不过好在我们可以利用 spring 的注解处理器，达到同样的目的。

1. 重写Sa-Token的注解处理器：
```java
@Configuration
public class SaTokenConfigure {
    @PostConstruct
    public void rewriteSaAnnotationStrategy() {
        SaAnnotationStrategy.instance.setAnnotationElementGetter((element, annotationClass) -> {
            return AnnotatedElementUtils.getMergedAnnotation(element, annotationClass);
        });
    }
}
```

2. 自定义一个注解：
```java
/**
 * 登录认证(User版)：只有登录之后才能进入该方法
 * <p> 可标注在函数、类上（效果等同于标注在此类的所有方法上）
 */
@SaCheckLogin(type = "user")
@Retention(RetentionPolicy.RUNTIME)
@Target({ ElementType.METHOD, ElementType.TYPE})
public @interface SaUserCheckLogin {

}
```

3. 接下来就可以使用我们的自定义注解了：
```java
// 使用 @SaUserCheckLogin 的效果等同于使用：@SaCheckLogin(type = "user")
@SaUserCheckLogin
@RequestMapping("info")
public String info() {
    return "查询用户信息";
}
```

注：其它注解 `@SaCheckRole("xxx")`、`@SaCheckPermission("xxx")` 同理，完整示例参考 Gitee 代码：注解合并。

## 8、同端多账号登录
假设我们不仅需要在后台同时集成两套账号，我们还需要在一个客户端同时登陆两套账号（业务场景举例：一个APP中可以同时登陆商家账号和用户账号）。

如果我们不做任何特殊处理的话，在客户端会发生token覆盖，新登录的token会覆盖掉旧登录的token从而导致旧登录失效。

具体表现大致为：在一个浏览器登录商家账号后，再登录用户账号，然后商家账号的登录状态就会自动失效。

那么如何解决这个问题？很简单，我们只要更改一下 `StpUserUtil` 的 `TokenName` 即可，参考示例如下：

```java
public class StpUserUtil {

    // 账号体系标识
    private static final String TYPE = "user";

    // 底层使用的 StpLogic 对象，通过 new StpLogic("user") 新建一个与 `StpUtil` 不同的token名称，防止冲突
    public static final StpLogic stpLogic = new StpLogic(TYPE) {
        @Override
        public String splicingKeyTokenName() {
            return super.splicingKeyTokenName() + "-" + TYPE;
        }
        // 同理可以按需重写一些其它方法...
    };

    // 其它代码 ...
}
```

再次调用 `StpUserUtil.login(10001)` 进行登录授权时，token的名称将不再是 `satoken`，而是我们重写后的 `satoken-user`，这样就不会再两次token写入同一个key而相互覆盖了。

## 9、不同体系不同sa-TokenConfig配置
如果自定义的`StpUserUtil`需要使用不同`SaTokenConfig`对象，也很简单，参考示例如下：

```java
@Configuration
public class SaTokenConfigure {

    @Autowired
    public void setConfig(SaTokenConfig config) {
        // 给 StpUtil 配置的 SaTokenConfig 配置参数对象
        StpUtil.setConfig(config);
        // 配置一些自定义参数
        config.setTokenName("satoken");
        config.setTimeout(2592000);
        config.setTokenStyle("uuid");
        // ... 更多配置 ...
        StpUtil.stpLogic.setConfig(config);


        // 给 StpUserUtil 配置的 SaTokenConfig 配置参数对象
        SaTokenConfig userConfig = new SaTokenConfig();
        // 配置一些自定义参数
        userConfig.setTokenName("satoken-user");
        userConfig.setTimeout(604800);
        userConfig.setTokenStyle("tik");
        // ... 更多配置 ...
        StpUserUtil.stpLogic.setConfig(userConfig);
    }

}
```

## 10、多账号体系混合鉴权
QQ群中经常有小伙伴问：在多账号体系下，怎么在 `SaInterceptor` 拦截器中给一个接口双重鉴权？

其实这个问题，主要看你的业务需求来决定，以后台 Admin 账号和前台 User 账号为例：

```java
@Override
public void addInterceptors(InterceptorRegistry registry) {
    // 注册 Sa-Token 拦截器
    registry.addInterceptor(new SaInterceptor(handler -> {

        // 如这个接口，要求客户端必须登录了后台 Admin 账号才能访问
        SaRouter.match("/art/getList").check(r -> StpUtil.checkLogin());

        // 如这个接口，要求客户端必须登录了前台 User 账号才能访问
        SaRouter.match("/goods/getList").check(r -> StpUserUtil.checkLogin());

        // 如这个接口，要求客户端同时登录了 Admin 和 User 账号，才能访问
        SaRouter.match("/comment/add").check(r -> {
            StpUtil.checkLogin();
            StpUserUtil.checkLogin();
        });

        // 如这个接口，要求客户端登录 Admin 或 User 账号任意一个，就能访问
        SaRouter.match("/index").check(r -> {
            if(StpUtil.isLogin() == false && StpUserUtil.isLogin() == false) {
                throw new NotLoginException("请登录", StpUtil.TYPE, NotLoginException.NOT_TOKEN);
            }
        });

    })).addPathPatterns("/**");
}
```

## 11、在一个接口里获取是哪个体系的账号正在登录
可以用如下的`getLoginType()`方法判断，哪个`StpLogic`正在登录了哪个体系：

```java
@RequestMapping("getInfo")
public SaResult getInfo() {
    // 判断哪个体系登录了
    if(StpUtil.isLogin()) {
        System.out.println("当前登录的是：admin 体系，账号id是：" + StpUtil.getLoginId());
    }
    if(StpUserUtil.isLogin()) {
        System.out.println("当前登录的是：user 体系，账号id是：" + StpUserUtil.getLoginId());
    }

    // 此处还能写成动态的形式：
    String loginType = StpUtil.getLoginType();
    System.out.println("当前登录的体系是：" + loginType);

    // 请注意此处可能出现的两种结果：
    // * 两个if都不成立：代表客户端两个账号体系都没有登录。
    // * 两个if同时成立：代表客户端在两个账号体系都登录了。
    // * 两个if只成立一个：代表客户端在哪个体系登录了。

    return SaResult.ok();
}
```

## 12、注意点：运行时不可更改 LoginType
在高并发访问时，更改账号类型可能会出现一些意想不到的问题：

```java
// 错误示例
StpUtil.stpLogic.loginType = "user";
StpUtil.login(10001);
StpUtil.stpLogic.loginType = "admin";
```

这种写法是错误的！LoginType 不应该在运行时更改，只能在项目启动时指定。一旦项目启动后再修改LoginType，就会造成线程安全问题和严重的逻辑错误。
