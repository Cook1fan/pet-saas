## 4. 核心业务逻辑与流程说明

### 4.1 多租户隔离流程

**方案：** 共享数据库 + 共享数据表 + tenant_id 字段隔离，MyBatis-Plus 多租户插件自动处理

**时序说明：**
1. 用户登录时，系统验证账号密码与租户权限
2. 登录成功后，将 tenant_id 存入 Sa-Token Session
3. 后续请求时，MyBatis-Plus 多租户插件自动从 Session 获取 tenant_id
4. 插件自动在所有 SQL 的 WHERE 条件中添加 `tenant_id = 当前租户ID`
5. 忽略租户隔离的表（如平台全局表）通过配置排除

**关键配置：**
- 所有业务表（除平台全局表）必须包含 `tenant_id bigint not null` 字段，添加索引
- 忽略租户隔离的表：`sys_platform_admin`、`sys_dict`、`sys_config`
- 平台超级管理员角色可访问所有租户数据

### 4.2 登录认证流程

**方案：** Sa-Token + 角色权限控制，多端登录隔离

**PC 端门店管理员登录时序：**
1. 前端调用 `/pc/login` 接口，传入 username、password、tenantId
2. 后端验证账号密码与租户权限：查询 `shop_admin` 表，匹配 username 和 tenantId
3. 密码校验通过后，调用 `StpUtil.login(admin.getId(), "pc")` 进行登录，指定设备类型为 pc
4. 将 tenant_id、admin_id、role 存入 Sa-Token Session
5. 返回 token 给前端
6. 后续请求前端在请求头携带 token，Sa-Token 自动校验
7. 权限校验通过注解 `@SaCheckRole("shop_admin")` 实现角色拦截

**多端登录配置：**
- PC 端 token 前缀：`pc_`
- 商家版小程序 token 前缀：`merchant_`
- C 端小程序 token 前缀：`user_`
- Token 有效期：30 天

### 4.3 开单收银流程

**开单收银时序：**
1. 店员选择商品/服务，可选择会员（可选）
2. 选择支付方式：微信/现金/储值/次卡
3. 系统校验：
   - 若使用储值：校验会员储值余额是否充足
   - 若使用次卡：校验会员次卡剩余次数是否充足、是否在有效期内
   - 若商品非服务类：校验商品库存是否充足
4. 创建订单：
   - 生成唯一订单号
   - 插入 `order_info` 表
   - 插入 `order_item` 表
5. 扣减库存（非服务类商品）
6. 扣减余额/次数（储值/次卡支付）
7. 生成流水记录：插入 `flow_record` 表
8. 若微信支付：调用微信支付 API，发起支付
9. 支付成功后，微信回调更新订单状态为已支付
10. 返回开单结果

### 4.4 拼团/秒杀流程

**核心目标：** 防止超卖、系统不崩溃、数据一致

**活动创建时序：**
1. 门店管理员创建拼团/秒杀活动，设置活动信息、库存
2. 系统保存活动信息到 `activity_info` 表
3. 系统将活动库存同步到 Redis：`set activity:stock:{activity_id} {库存数}`
4. 生成活动小程序码

**用户下单时序：**
1. C 端用户点击活动，请求下单
2. Redis 预扣库存：`decrement activity:stock:{activity_id}`
3. 若库存 < 0：回补库存，返回「活动库存不足」
4. 获取 Redisson 分布式锁：`order:user:{userId}:activity:{activityId}`
5. 若获取锁失败：返回「请勿重复下单」
6. 创建活动订单：插入 `activity_order` 表
7. 创建主订单：插入 `order_info`、`order_item` 表
8. 调用微信支付发起支付
9. 支付成功后，释放分布式锁
10. 通过 RabbitMQ 异步发送通知、更新销量

### 4.5 支付与对账流程

**资金流向：** 客户支付直接调用门店自有的微信商户号，资金 T+1 自动提现到门店银行卡，平台不碰资金

**支付回调处理时序：**
1. 微信支付成功后，回调后端接口
2. 后端验证回调签名，确保请求来自微信
3. 查询系统订单，验证金额是否一致
4. 若订单已处理：直接返回成功（幂等性保证）
5. 更新订单状态为已支付
6. 生成流水记录
7. 返回成功给微信

**自动对账时序：**
1. 每日凌晨定时任务触发
2. 遍历所有门店，获取门店微信商户号配置
3. 调用微信支付账单 API，下载门店前一日的交易账单
4. 遍历账单记录，与系统 `flow_record` 表通过微信支付单号匹配
5. 金额、状态一致的：标记为「已对账」
6. 金额/状态不一致的：标记为「异常」，触发告警
7. 保存对账结果

### 4.6 AI 功能调用流程

**AI 文案生成时序：**
1. 门店管理员选择文案类型（朋友圈/活动海报/客户群问候），输入关键词
2. 检查门店 AI 调用额度：Redis 记录 `ai:limit:{tenantId}:{date}`
3. 若超出免费额度（50次/天）：返回「今日AI调用额度已用完」
4. 构建 Prompt，包含门店名称、关键词、文案要求
5. 调用 Spring AI Alibaba 对接豆包大模型 API
6. 模型返回 3 条备选文案
7. 记录 AI 调用到 `ai_call_record` 表
8. 返回文案给前端

**AI 客服问答时序：**
1. C 端用户输入问题
2. 检查门店 AI 调用额度
3. 优先匹配门店配置的常见问题
4. 若匹配到：直接返回配置的标准答案
5. 若未匹配：构建 Prompt，包含门店信息、用户问题
6. 调用豆包大模型 API
7. 记录 AI 调用
8. 返回回答给前端

### 4.7 小程序登录流程

#### 4.7.1 商家版小程序登录时序

1. 商家在小程序输入账号、密码、租户 ID
2. 后端验证租户是否存在且状态正常
3. 根据账号和租户 ID 查询 `shop_admin` 表
4. BCrypt 校验密码是否正确
5. 验证账号状态是否启用
6. 调用 `StpUtil.login(admin.getId(), "merchant")` 登录，指定设备类型为 `merchant`
7. 将 tenant_id、admin_id、role 存入 Sa-Token Session
8. 返回 token（前缀为 `merchant_`）和管理员信息

#### 4.7.2 C 端微信小程序登录时序

1. 前端调用 `wx.login()` 获取临时 code
2. 前端调用 `/api/user/wxLogin`，传入 code 和 tenantId（扫码进入时传入）
3. 后端使用 code 调用微信 `auth.code2Session` 接口，换取 openid 和 session_key
4. 根据 openid 查询是否已存在会员：
   - 不存在：创建新会员，openid 存入 `member` 表，同时创建会员账户
   - 存在：获取已有会员信息
5. 如果传入了 tenantId，检查 `member_shop_bind` 表是否已绑定该门店：
   - 未绑定：创建绑定记录
   - 已绑定：更新最后访问时间
6. 调用 `StpUtil.login(member.getId(), "user")` 登录，指定设备类型为 `user`
7. 将 tenantId、memberId、openid 存入 Sa-Token Session
8. 返回 token（前缀为 `user_`）和会员信息

**多门店切换逻辑：**
- 用户在「我的门店」中查看所有已绑定的门店列表
- 点击切换门店时，更新 Session 中的 tenantId 为新门店的 ID
- 重新查询该门店下的会员信息（一个 openid 在不同门店下是不同的会员记录）

### 4.8 核销码流程

#### 4.8.1 生成核销码时序（C 端）

1. C 端用户选择要核销的次卡，请求生成核销码
2. 验证会员次卡是否存在且属于当前会员
3. 验证次卡剩余次数是否大于 0
4. 验证次卡是否在有效期内
5. 将该会员次卡之前未使用的核销码标记为已失效（status=2）
6. 生成 6 位随机数字作为新核销码
7. 创建核销记录：插入 `verify_code_record` 表，status=0（未使用）
8. 设置过期时间为当前时间 +5 分钟
9. 返回核销码和过期时间

#### 4.8.2 次卡核销时序（商家版）

1. 商家在小程序扫描或输入核销码
2. 根据核销码查询 `verify_code_record` 表
3. 验证核销码是否存在
4. 验证核销码状态：status 必须为 0（未使用）
5. 验证核销码是否过期：当前时间 < expire_time
6. 查询关联的会员次卡信息 `member_card`
7. 验证次卡剩余次数是否大于 0
8. 验证次卡是否在有效期内
9. 扣减次卡剩余次数：`remain_times = remain_times - 1`
10. 更新核销码记录：status=1（已使用），记录 verify_time 和 verify_user_id
11. 创建核销流水记录（可选，插入 `flow_record` 表）
12. 返回核销结果：次卡名称、会员姓名、剩余次数、过期时间

### 4.9 C 端活动下单流程

**活动下单时序：**
1. C 端用户选择活动，确认购买数量
2. 验证活动是否存在且状态为进行中（status=1）
3. 验证活动时间是否在有效范围内：start_time <= 当前时间 <= end_time
4. 验证购买数量是否超过限购数量：num <= limit_num
5. Redis 预扣库存：`decrement activity:stock:{activityId}`
6. 若库存 < 0：回补库存（`increment activity:stock:{activityId}`），返回「活动库存不足」
7. 获取 Redisson 分布式锁：`order:user:{userId}:activity:{activityId}`
8. 若获取锁失败：返回「请勿重复下单」
9. 创建活动订单：插入 `activity_order` 表，status=0（待支付）
10. 创建主订单：插入 `order_info` 表、`order_item` 表，pay_status=0（待支付）
11. 调用微信支付统一下单 API，获取支付参数
12. 返回订单信息和微信支付参数给前端
13. 前端调起微信支付
14. 支付成功后，微信回调后端接口
15. 后端验证回调签名，更新订单状态为已支付
16. 释放分布式锁
17. 异步更新活动销量、发送通知
