## 一、背景与问题

### 1.1 当前代码库存在的问题

| 序号 | 问题 | 优先级 | 说明 |
|-----|------|--------|------|
| 1 | 库存扣减时机错误 | P0 | 微信支付订单在待支付状态就扣减了实际库存 |
| 2 | 预留库存字段未正确使用 | P1 | `GoodsSku.reservedStock` 字段存在但流程未正确利用 |
| 3 | 缺少完整订单状态管理 | P1 | 只有支付状态，没有订单全生命周期状态流转 |
| 4 | 微信支付对接不完整 | P0 | `WechatPayServiceImpl` 只是模拟实现 |
| 5 | 缺少订单超时取消机制 | P1 | 未支付订单没有自动释放库存的定时任务 |
| 6 | 缺少订单操作日志 | P2 | 没有记录订单状态变更历史 |
| 7 | 未区分业务场景 | P0 | PC端开单收银和小程序下单的库存扣减时机应该不同 |
| 8 | 定时任务未考虑多实例 | P1 | 简单的 `@Scheduled` 在多实例/微服务环境下会有问题 |

## 三、核心业务流程设计

### 3.1 两种业务场景的库存管理

#### 库存字段说明

| 字段 | 说明 |
|-----|------|
| stock | 可用库存（可对外售卖的数量） |
| reservedStock | 预留库存（已锁定的数量） |

#### 场景一：PC端开单收银（到店即时交付）

**特点：** 客户到店，支付后直接拿走商品

**库存流转：**

| 支付方式 | 下单时 | 支付成功 |
|---------|--------|----------|
| 现金/储值/次卡 | 直接扣减 stock | - |
| 微信支付 | stock→reservedStock | 扣减 reservedStock |

**订单状态流转：**
```
UNPAID(待支付)
    ├─→ SHIPPED(已发货/已完成)  [支付回调，同时扣库存]
    │     └─→ REFUNDED(已退款)
    └─→ CANCELLED(已取消)
```

#### 场景二：小程序下单（电商发货）

**特点：** 用户在线支付后，店家需要时间打包、发货

**库存流转：**

| 步骤 | 库存操作 | 订单状态 |
|-----|---------|----------|
| 下单 | stock→reservedStock | UNPAID |
| 支付成功 | 不扣库存 | PAID |
| 店家发货 | 扣减 reservedStock | SHIPPED |

**订单状态流转：**
```
UNPAID(待支付)
    ├─→ PAID(已支付)  [支付回调，不扣库存]
    │     ├─→ SHIPPED(已发货)  [店家点击发货，扣库存]
    │     │     └─→ COMPLETED(已完成)
    │     └─→ REFUNDED(已退款)
    └─→ CANCELLED(已取消)
```

## 四、核心实现细节

### 4.1 库存管理优化

**GoodsSkuService 新增方法：**

```java
/**
 * 预占库存：从可用库存(stock)移到预留库存(reservedStock)
 * 下单时调用（微信支付场景）
 */
boolean reserveStock(Long skuId, int num);

/**
 * 释放预占库存：从预留库存移回可用库存
 * 订单取消/超时时调用
 */
boolean releaseReservedStock(Long skuId, int num);

/**
 * 确认扣减：扣减预留库存
 * 支付成功后调用（PC端开单收银）或发货时调用（小程序下单）
 */
boolean confirmDeductReservedStock(Long skuId, int num);

/**
 * 直接扣减可用库存
 * 即时交付场景用（现金/储值/次卡支付）
 */
boolean deductStockDirect(Long skuId, int num);
```

### 4.2 PC端开单收银流程重构

**即时支付模式（现金、储值、次卡）：**
1. 验证库存
2. 创建订单（orderSource=PC，orderStatus=SHIPPED）
3. 直接扣减可用库存（stock -= num）
4. 记录库存变动和流水

**扫码支付模式（微信支付）：**
1. 验证库存
2. 预占库存（stock → reservedStock）
3. 创建订单（orderSource=PC，orderStatus=UNPAID，expireTime=now+15分钟）
4. 调用微信支付统一下单API
5. 返回支付参数给前端

### 4.3 微信支付回调处理

**关键步骤：**
1. 验证回调签名
2. 幂等性检查（Redis Key: `order:pay:callback:{orderNo}`）
3. 查询订单，验证金额
4. 判断订单来源：
   - **PC端开单收银**：确认扣减预留库存，订单状态直接到 SHIPPED（已发货/已完成）
   - **小程序下单**：不扣库存，订单状态只到 PAID（已支付）
5. 生成流水记录
6. 返回成功响应

### 4.4 小程序下单流程（电商发货）

**下单：**
1. 验证库存
2. 预占库存（stock → reservedStock）
3. 创建订单（orderSource=USER_MINIAPP，orderStatus=UNPAID，expireTime=now+30分钟）
4. 调用微信支付统一下单API
5. 返回支付参数

**支付成功回调：**
1. 更新订单状态到 PAID（已支付）
2. **不扣库存**，等待店家发货

**店家点击发货：**
1. 验证订单状态是 PAID
2. 确认扣减预留库存（reservedStock -= num）
3. 更新订单状态到 SHIPPED（已发货）
4. 记录库存变动

### 4.5 订单超时取消定时任务（支持多实例/微服务）

**方案：Redisson 分布式锁 + 订单状态机幂等性保证**

```java
@Slf4j
@Component
@RequiredArgsConstructor
public class OrderTimeoutCancelJob {

    private final OrderService orderService;
    private final RedissonClient redissonClient;

    /**
     * 每分钟执行一次，取消超时未支付订单
     * 使用 Redisson 分布式锁保证多实例环境下只有一个实例执行
     */
    @Scheduled(cron = "0 * * * * ?")
    public void cancelTimeoutOrders() {
        String lockKey = "job:order:timeout:cancel";
        RLock lock = redissonClient.getLock(lockKey);

        // 尝试获取锁，等待时间3秒，锁持有时间5分钟
        try {
            if (!lock.tryLock(3, 300, TimeUnit.SECONDS)) {
                log.info("其他实例正在执行订单超时取消任务，跳过本次执行");
                return;
            }
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            log.error("获取分布式锁被中断", e);
            return;
        }

        try {
            log.info("开始执行订单超时取消任务");
            this.doCancelTimeoutOrders();
            log.info("订单超时取消任务执行完成");
        } catch (Exception e) {
            log.error("订单超时取消任务执行异常", e);
        } finally {
            if (lock.isHeldByCurrentThread()) {
                lock.unlock();
            }
        }
    }

    @Transactional(rollbackFor = Exception.class)
    protected void doCancelTimeoutOrders() {
        // 每次处理100条，分批处理避免长事务
        int pageSize = 100;
        int processedCount = 0;

        while (true) {
            List<OrderInfo> timeoutOrders = orderService.listTimeoutUnpaidOrders(pageSize);
            if (CollectionUtils.isEmpty(timeoutOrders)) {
                break;
            }

            for (OrderInfo order : timeoutOrders) {
                try {
                    // 取消订单（内部有状态机校验，保证幂等）
                    orderService.cancelOrder(order.getId(), "订单超时自动取消", true);
                    processedCount++;
                } catch (Exception e) {
                    log.error("取消订单失败，orderId={}", order.getId(), e);
                }
            }

            if (timeoutOrders.size() < pageSize) {
                break;
            }
        }

        log.info("本次订单超时取消任务处理数量：{}", processedCount);
    }
}
```

**多实例环境下的三层保证：**

1. **第一层：Redisson 分布式锁**
   - 同一时间只有一个实例能执行任务
   - 锁 Key: `job:order:timeout:cancel`
   - 锁持有时间：5分钟（避免死锁）

2. **第二层：分批查询 + 状态机校验**
   - 每次只查询100条待处理订单
   - `cancelOrder()` 内部校验订单状态，只有 UNPAID 才能取消

3. **第三层：数据库乐观锁**
   - 更新订单状态时使用 `version` 字段
   - 防止并发更新导致的数据不一致

**微服务拆分后的演进方案：**

如果将来拆分为独立的订单服务，可以考虑：
- **XXL-Job / Elastic-Job**：分布式调度中心
- **事件驱动**：订单创建时发送延迟消息到 MQ，到期自动取消
