# 宠物门店 SaaS 端到端测试方案

## 1. 测试框架选型

### 1.1 PC 端测试框架：Playwright

**选择理由**：
- 支持多浏览器（Chrome、Firefox、Safari）
- 自动等待机制，减少 flaky 测试
- 强大的调试工具（Playwright Inspector）
- 支持多端设备模拟
- 内置截图、录制功能
- TypeScript 支持完善

**核心特性**：
- 并行测试执行
- 测试失败自动重跑
- 自动生成测试报告
- Trace 完整记录执行过程

### 1.2 小程序端测试框架：UniApp Automator + 微信开发者工具

**选择理由**：
- 官方维护，与 UniApp 深度集成
- 支持微信开发者工具自动化
- 提供真实设备测试能力
- 支持真机调试

### 1.3 API 集成测试：JUnit 5 + Spring Boot Test + Postman

**选择理由**：
- Spring 官方支持，集成良好
- 支持 MockMvc 快速测试 API
- Postman 提供直观的 API 测试和文档
- 支持 Newman 命令行集成

### 1.4 性能测试：JMeter + Artillery

**选择理由**：
- JMeter 成熟稳定，支持各类协议
- Artillery 轻量级，配置简单
- 支持真实场景模拟

## 2. 测试用例设计策略

### 2.1 测试用例组织原则

**按业务流程组织**：
- ✅ 完整的用户旅程
- ✅ 关键业务闭环
- ✅ 异常流程分支

**测试用例优先级**：
- **P0**：核心流程，阻断性缺陷，必须覆盖
- **P1**：重要功能，严重缺陷，重点覆盖
- **P2**：一般功能，建议类缺陷，尽量覆盖

### 2.2 核心业务流程测试

#### 流程1：拼团活动全流程

**商家端（PC）**：
1. 登录商家后台
2. 创建拼团活动
3. 配置活动规则
4. 发布活动
5. 查看活动数据

**小程序端（用户）**：
1. 登录小程序
2. 浏览活动列表
3. 查看活动详情
4. 发起/加入拼团
5. 支付
6. 等待成团
7. 查看订单状态

**后端**：
1. 库存扣减
2. 支付处理
3. 团状态管理
4. 定时任务处理
5. 退款处理

#### 流程2：储蓄卡次卡全流程

**商家端（PC）**：
1. 创建卡规则
2. 设置有效期
3. 设置使用限制

**小程序端（用户）**：
1. 浏览卡商品
2. 购买卡
3. 查看卡余额
4. 核销使用
5. 查看消费记录

### 2.3 异常场景测试

**网络异常**：
- 网络断开
- 网络超时
- 弱网环境

**业务异常**：
- 库存不足
- 拼团已满
- 卡已过期
- 余额不足

**系统异常**：
- 数据库宕机
- Redis 不可用
- 第三方服务不可用

### 2.4 边界场景测试

- 库存为 0
- 时间边界（活动即将开始/结束）
- 金额边界（最小/最大金额）
- 人数边界（成团人数临界点）

- 🚨 致命：系统崩溃、数据丢失

- 🟡 一般：功能缺陷但不阻断
- 🟢 建议：体验优化建议

**缺陷跟踪工具**：
- GitHub Issues
- 自建缺陷管理系统

## 6. 最佳实践与规范

### 6.1 测试脚本开发规范

**选择器规范**：
- 优先使用 `data-testid` 属性
- 其次使用稳定的定位方式
- 避免使用动态属性或位置选择器

**等待策略**：
- 使用显式等待（waitForSelector）
- 避免硬编码 sleep
- 使用 Playwright 的智能等待

**断言策略**：
- 断言业务逻辑，不只断言UI
- 每个测试只断言一个主要结果
- 使用有意义的断言消息

**测试隔离**：
- 每个测试独立运行
- 测试前后清理环境
- 使用独立的测试数据

### 6.2 测试数据管理

**测试数据生成**：
- 使用 Faker 等库生成
- 数据工厂模式
- 边界值和特殊字符测试

**数据隔离**：
- 使用独立的测试租户
- 测试完成后自动清理
- 不影响生产数据

### 6.3 测试维护与优化

**定期更新**：
- 业务变更同步更新测试用例
- 定期检查失效的选择器
- 删除过时的测试

**性能优化**：
- 并行执行测试
- 按需初始化数据
- 合理使用缓存

**调试技巧**：
- 使用 Playwright Inspector
- 开启 trace 记录
- 视频回放失败测试
