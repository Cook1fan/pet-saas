# 宠物SaaS项目 - 测试规范总则
## 文档信息
- 所属目录：docs/05-测试规范/
- 适用角色：@Architect、@Backend、@Frontend、@MiniApp、@Tester
- 定位：项目测试体系总章程，所有测试活动必须遵循本规范
- 生效原则：**文档驱动、规范先行、测试左移、分层负责**

---

# 1 测试分层与责任归属
## 1.1 单元测试（Unit Test）
### 责任主体
@Backend / @Frontend / @MiniApp **开发自治，谁开发谁负责**

### 覆盖范围
- 后端：Service、Utils、公共工具类、核心业务算法、枚举校验
- 管理后台前端：公共组件、自定义Hook、工具函数、常量校验逻辑
- 微信小程序：通用方法、数据格式化、表单校验、业务工具函数

### 准入要求
代码提交前必须：
1. 单测核心分支覆盖通过
2. 无失败用例、无跳过关键用例
3. @Architect / @Tester 有权评审覆盖率与用例质量

### 产出物
单元测试代码跟随业务代码同目录维护，不单独归档文档。

---

## 1.2 集成测试（Integration Test）
### 责任主体
**@Tester 主导设计 & 执行**，@Backend 配合支撑

### 覆盖范围
- 多接口串行/并行链路调用
- 数据库、缓存、中间件联动交互
- 跨模块、跨服务业务流程连通性
- 接口参数合法性、异常参数、权限控制、事务边界

### 配合职责
- @Backend：提供测试环境、初始化基础数据、协助排查接口底层报错、修复缺陷
- @Tester：编写集成测试用例、接口自动化脚本、回归复测、记录缺陷

### 产出物
集成测试用例、接口自动化脚本、集成测试报告，归档至当前目录。

---

## 1.3 端到端 E2E 全链路测试
### 责任主体
**@Tester 全权负责**

### 覆盖范围
1. 商家管理后台 PC 端：完整业务操作闭环
2. 微信小程序 C 端：用户从进店、会员、预约、下单、支付、核销、退款全链路
3. 多门店、多角色、权限隔离、边界场景、异常流程、机型/微信版本兼容
4. 界面UI、交互跳转、文案展示、弹窗异常、白屏卡顿

### 配合职责
- @Frontend / @MiniApp：保证页面元素稳定、关键节点预留测试埋点、协助定位UI/适配问题
- 前端/小程序**不负责编写 E2E 用例与自动化脚本**

### 工具约定
可选：Playwright / Cypress / 小程序官方自动化工具
统一脚本存放：docs/05-测试规范/e2e-script/

### 产出物
E2E 业务流程用例、自动化脚本、全链路测试报告、兼容性测试报告。

# 2 提测与验收流程
## 2.1 提测前置条件
必须同时满足：
1. 对应模块 **PRD、模块设计、API契约** 文档已定稿
2. 开发编码完成、**单元测试全部通过**
3. 本地/测试环境部署正常、基础初始化数据齐全
4. 无阻塞性未完成功能

## 2.2 标准提测话术
开发统一格式：
> @Tester 【XX模块】已开发完成、单元测试自测通过，环境已部署，请安排集成测试 + E2E 全链路测试。

## 2.3 测试准出标准
1. 所有核心流程用例 100% 通过
2. 无致命/严重级缺陷
3. 一般缺陷有修复计划或已闭环
4. 回归测试全部通过
5. 输出正式测试报告，可进入联调/发版阶段

---

# 3 缺陷管理规范
1. 缺陷统一记录、分级：致命 > 严重 > 一般 > 建议
2. 致命/严重缺陷：**立即阻断、优先修复、复测通过才可继续**
3. 缺陷修复后由 @Tester 负责回归验证闭环
4. 所有重大缺陷复盘记录归档至本目录下缺陷复盘文档

# 5 约束铁律
1. 严禁跳过单元测试直接提测
2. 严禁无文档依据自行降低测试验收标准
3. 集成、E2E 测试必须由 @Tester 主导，开发不得替代包办
4. 测试范围、流程标准有争议，由 @Architect 仲裁并更新本规范文档
